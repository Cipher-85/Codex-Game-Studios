# Risk Register - Phase 3

Phase 4 status: complete.

This register covers non-1:1 Claude-to-Codex mappings identified through Phase 3. Later phases should add implementation-specific risks discovered by validation design.

| ID | Risk | Why it matters | Recommended Codex-native substitute | Verification strategy | Status |
|---|---|---|---|---|---|
| R1 | `AGENTS.md` has no verified `@import` equivalent | Upstream root `CLAUDE.md` imports several docs; relying on imports would silently drop project guidance. | Inline critical root guidance; put detailed references under `docs/`; split path rules into nested `AGENTS.md`. | Grep generated `AGENTS.md` for `@`; run instruction-size check against `project_doc_max_bytes`; manual prompt-input inspection if available. | Open for implementation |
| R2 | Existing target `AGENTS.md` collision | Many game projects may already have Codex instructions; blind overwrite would break unrelated workflows. | Marker-managed CCGS block in Markdown files; refuse ambiguous overwrites; manifest ownership. | Install fixture with preexisting `AGENTS.md`; assert non-CCGS content unchanged after install/uninstall. | Architecture decision made |
| R3 | Upstream `.claude/rules/*.md` are path-scoped, while Codex `.rules` are command approval policies | Confusing the two would either fail to load standards or create nonsensical command rules. | Convert upstream path rules to nested `AGENTS.md`; use `.codex/rules/settings.rules` only for command approval/deny rules. | Count 11 upstream rules; assert 11 nested instruction targets; parse `.codex/rules/settings.rules` separately. | Architecture decision made |
| R4 | Permission profiles do not compose with legacy `sandbox_mode` | GLM plan mixed both; Codex may ignore the intended profile or use older sandbox behavior. | Use `default_permissions = "game_studios"` and `[permissions.game_studios] extends = ":workspace"`; avoid `sandbox_mode` in distributable config. | `codex --strict-config`; static scan for both `default_permissions` and `sandbox_mode` in generated project config. | Architecture decision made |
| R5 | Command `prefix_rules` are not `config.toml` entries | A plan that writes command rules into config is non-executable. | Put command policies in `.codex/rules/settings.rules` using `prefix_rule(...)` syntax. | Parser/lint for `.rules`; inline `match`/`not_match` cases; smoke blocked commands in a fixture. | Architecture decision made |
| R6 | Project `.codex/` layers require trust | Hooks, project config, and rules may not run after install, causing false parity. | Installer docs and audit must detect trusted/untrusted state; provide explicit trust step. | Fixture run in untrusted/trusted modes; assert audit reports inactive hooks/config when untrusted. | Open for validation |
| R7 | Claude `Notification` hook has no Codex lifecycle event | Upstream `notify.sh` cannot be wired 1:1 as a project hook, but dropping notification behavior entirely would be a parity miss. | Do not install `notify.sh` as a project hook. Document native Codex notifications, including user-level `notify`, `[tui].notifications`, `[tui].notification_method`, and `[tui].notification_condition`; note that project config cannot set `notify`. | Assert `Notification` absent from `.codex/hooks.json`; docs mention native notification setup and project-local limitation. | Architecture decision revised |
| R8 | Claude scriptable `statusLine.command` has no verified Codex project script footer replacement | Codex supports `tui.status_line`, but verified docs expose built-in footer item IDs, not arbitrary project shell rendering. A literal footer `Stage:` item cannot be guaranteed directly in the footer. | Configure Codex `[tui].status_line` for built-in model/context/current-dir visibility; preserve game-stage/review/session breadcrumb through `studio-status-on-start.sh` and `studio-status`; document actual footer Stage parity as blocked until Codex exposes a documented custom footer item. | Config validation for supported status_line IDs; hook fixture using `production/session-state/active.md`; skill smoke reads same block; no fake custom footer key. | Blocked pending Codex support |
| R9 | `PostToolUse` cannot prevent an already-applied edit | Upstream `validate-assets.sh` intended safety may become only advisory if naively ported. | Move blocking checks to `PreToolUse` where input shape permits; keep `PostToolUse` for advisory checks. | apply_patch hook fixture with valid/invalid asset JSON; assert deny shape or nonzero block behavior. | Open for implementation |
| R10 | Codex edit tool names differ from Claude `Write|Edit` | Hook matchers may never fire if copied literally. | Use Codex-supported matcher values such as `apply_patch`, with compatibility aliases only after verification. | Hook fixture for `apply_patch`; grep hooks for literal Claude-only `Write|Edit` assumptions. | Open for implementation |
| R11 | Claude skill frontmatter has fields Codex does not enforce | Keeping `allowed-tools`, `model`, `argument-hint`, or `user-invocable` as if enforced creates false safety. | Keep only required Codex metadata; move Claude metadata to body notes; enforce safety through config/rules/hooks. | Skill metadata lint; scan for unsupported enforcement claims. | Architecture decision made |
| R12 | Claude `AskUserQuestion` has no verified direct Codex equivalent | Structured choice gates are central to CCGS collaboration behavior. | Numbered natural-language options by default; optional MCP elicitation only if a real MCP server is introduced. | Grep all skills for `AskUserQuestion`; review rewritten choices; interactive smoke for `start` and `gate-check`. | Open for implementation |
| R13 | Claude `Task` delegation is not syntax-compatible with Codex | Role delegation is core behavioral parity. | Rewrite to Codex subagent delegation by exact upstream role name, backed by `.codex/agents/<role>.toml`. Preserve names such as `producer`, `writer`, and `game-designer`. | Grep skills for `Task`; verify each referenced role maps to a TOML; sample subagent spawn smoke. | Open for implementation |
| R14 | Claude agent-pinned `skills` are not verified as exact Codex `skills.config` lists | Overstated parity could cause agents to lose expected workflow affinities. | Preserve affinity in `developer_instructions`; use skill config only after schema validation. | Validate custom-agent TOML schema; sample agent prompts for pinned workflows. | Open for Phase 4 validation |
| R15 | Claude per-agent `tools` and `disallowedTools` have no exact Codex equivalent | Some agents intentionally cannot run Bash; imprecise mapping can weaken safety. | Use permission profile, command `.rules`, hooks, and explicit agent instructions; mark partial. | Security static review; forbidden command smoke; inspect all 15 `disallowedTools: Bash` agents. | Open for implementation |
| R16 | Claude `maxTurns` has no verified per-agent Codex equivalent | Long-running or recursive delegation may differ from upstream. | Use `agents.max_threads`, `agents.max_depth`, timeouts where verified, and instruction-level limits. | Subagent config validation; smoke delegation does not fan out indefinitely. | Open for validation |
| R17 | Claude `memory` metadata has no direct documented Codex per-agent binding | User/project memory semantics may be lost if collapsed into generic prose, while writing global Codex memories would break repo self-containment and could pollute user state. | Preserve every upstream `memory: user` and `memory: project` declaration; create `.codex/agent-memory/<agent>/MEMORY.md` for all 17 memory-scoped agents; bind each custom agent to read its memory file in `developer_instructions`; do not write `~/.codex/memories` during install. | Manifest includes 17 memory targets; agent TOML binding scan; lead-programmer explicit memory file content rewrite scan; install audit checks no writes outside target. | Architecture decision revised |
| R18 | Worktree isolation is not automatic for Codex subagents | `prototyper`, `prototype`, and `vertical-slice` expect isolation-like behavior. | Explicit worktree instructions plus optional helper script under `.codex/lib/worktree.sh`. | Worktree smoke test if helper exists; otherwise manual validation checklist. | Open for implementation |
| R19 | Repo-local custom slash commands are not verified | `/start` muscle memory cannot be committed as project slash commands. | Use skills: `/skills`, `$start`, or implicit selection; docs can mention built-in `/plan` only for planning. | Grep runtime files for bare `/start`, `/brainstorm`, etc.; docs invocation check. | Architecture decision revised |
| R20 | Model slugs are high-drift | Hardcoded unavailable models would break subagents. | Store tier mapping in `.codex/models.toml`; validate against `codex debug models`. | Model slug lint in audit; current slugs verified in Phase 2. | Open for validation |
| R21 | Local cached plugin can mislead implementation | Cache contains a `codex-game-studios` plugin, but workspace marketplace is invalid. | Treat cache as non-authoritative; build from upstream inventory and corrected plan only. | `codex plugin list` failure documented; no implementation steps read plugin cache as source. | Architecture decision made |
| R22 | Testing framework missing `vertical-slice` spec upstream | A naive completeness check may falsely blame the Codex port or miss upstream gap. | Preserve the upstream inventory as an inherited gap and add a clearly labeled Codex-specific `vertical-slice` spec covering PROCEED/PIVOT/KILL. | Framework catalog includes `vertical-slice`; manifest distinguishes upstream inventory from Codex-added target coverage. | Mitigated by Codex-added spec |
| R23 | Shared neutral state can be concurrently edited by Claude and Codex | Coexistence needs shared artifacts, but simultaneous sessions could conflict. | Preserve same paths but rely on git/status checks, session-state logs, and review gates; do not create separate stage/review files. | Coexistence smoke with alternating Claude/Codex-style writes; git diff review. | Open for validation |
| R24 | Installer/uninstaller can accidentally remove shared state | Shared `production/` and `design/` are not Codex-owned even when Codex writes them. | Manifest owns only Codex files and marker blocks; uninstaller never removes shared neutral dirs. | Install/uninstall fixture; hash `.claude/**`, `CLAUDE.md`, and shared state before/after. | Architecture decision made |
| R25 | Existing `.codex/hooks.json` or `.codex/config.toml` collision | JSON/TOML cannot be marker-managed as safely as Markdown. | If absent, create. If existing and unowned, refuse by default and provide manual merge instructions; `--force` backs up first. | Fixture with preexisting config/hooks; assert refusal and no modification. | Architecture decision made |
| R26 | Runtime files may retain `.claude/` references | Violates self-contained requirement and can break when Claude assets are absent. | Rewrite runtime references to neutral docs; keep `.claude` only in attribution/migration provenance. | `audit no-claude-refs` with strict allowlist. | Open for validation |
| R27 | Skill names may collide with other active skills | Using upstream names under `.agents/skills` preserves CCGS ergonomics but can conflict with same-name user, plugin, or system skills. Codex docs do not say project skills suppress duplicate skill names. | Use direct unprefixed `.agents/skills/<name>/SKILL.md` folders with `name: <name>` for parity; add collision audit that reports competing paths and requires disable-by-path or explicit acceptance of selector ambiguity. Current verified active context after stale plugin removal has no active default `prototype` skill. | Fixture with existing `start` and `prototype` skills; assert installer/audit reports collisions with paths and does not silently rename CCGS skills. | Architecture decision revised |
| R28 | Hook scripts may assume Claude stdin JSON field names | Naive shell port will parse the wrong data and silently skip checks. | Rewrite scripts against Codex hook fields: `hook_event_name`, `tool_name`, `tool_input`, `agent_type`, `cwd`, etc. | Fixture JSON tests per hook; shellcheck/static parse where available. | Open for implementation |
| R29 | Non-interactive smoke workflows may be blocked by approvals or hook trust | CI validation could be flaky or hang. | Separate static/headless audit from manual/interactive smoke; document trust/approval prerequisites. | `audit smoke-headless` vs `audit smoke-interactive` split. | Architecture decision made |
| R30 | MIT attribution could be lost during path rewrites | Legal/provenance requirement. | Preserve upstream MIT license and add `ATTRIBUTION.md` plus migration docs. | License/attribution file existence and content check. | Open for implementation |
| R31 | Nested namespace directories for repo skills or custom agents may not be discovered | Official docs describe direct skill folders under `.agents/skills` and standalone TOML files under `.codex/agents`; a nested namespace folder could silently make the port unavailable. | Use direct paths: `.agents/skills/<skill>/SKILL.md` for skills and `.codex/agents/<agent>.toml` for custom agents; reserve any plugin display namespace for an optional packaged distribution only. | Static path-shape lint; run Codex skill/agent discovery smoke after install; grep generated paths for nested namespace layout. | Architecture decision made |
| R32 | Validator depends on undeclared Python packages | Prior local validation attempts can fail if `yaml` or other non-stdlib modules are unavailable, turning the acceptance gate into an environment problem. | Use Python 3.11+ standard library only: `json`, `hashlib`, `tomllib`, and a small frontmatter parser for simple metadata. | Run `audit all` in a clean fixture without PyYAML installed; report setup errors separately from validation failures. | Architecture decision made |
| R33 | Codex CLI smoke commands may depend on trust, auth, user config, or model access | A model-running smoke test can fail for environmental reasons even when static port assets are valid. | Keep `audit all` static/headless by default; put `codex debug prompt-input` and `codex exec` checks behind `smoke-interactive`. | CI runs `audit all`; release checklist records whether interactive smoke was run or skipped with reason. | Architecture decision made |
| R34 | `.rules` syntax can drift without an official standalone parser | A hand-written parser may accept rules Codex rejects or reject rules Codex accepts. | Use a conservative static lint plus command example fixtures; optionally verify with Codex CLI in an interactive/trusted fixture. | Positive/negative command examples; `codex --strict-config` prompt-input smoke where available. | Open for implementation |
| R35 | Manifest count checks can pass while target behavior is incomplete | A copied file can exist but still contain stale Claude commands, paths, or unsupported metadata. | Pair manifest coverage with rewrite scans and workflow smoke checks for critical lifecycle skills. | `audit manifest` plus `audit runtime`, `audit skills`, and `audit smoke-headless`. | Open for implementation |
| R36 | Hook fixture inputs may not cover every real Codex payload shape | Hook scripts could pass synthetic fixtures but fail during actual tool use. | Base fixtures on official hook fields and captured real payloads when available; keep hook scripts defensive around missing fields. | Fixture tests plus optional trusted-project live hook smoke for Bash and apply_patch events. | Open for implementation |
